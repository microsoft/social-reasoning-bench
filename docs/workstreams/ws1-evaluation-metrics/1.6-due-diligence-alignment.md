# 1.6 Due Diligence Alignment

## Definition & Architectural Decision

**Decision:** Due diligence = count of domain-specific "unit of work" tool calls. Only effort actions count, not conclusions. Calendar: RequestMeeting + ReplyMeeting(COUNTER). Marketplace: MakeOffer. Form-filling: Search* + Read*.

Due diligence measures whether the agent put in sufficient effort before reaching a conclusion. The metric is a count of domain-appropriate effort actions (tool calls that represent substantive work), excluding conclusion actions (EndConversation, AcceptOffer, fill_form) and passive actions (Wait, GetMessages, ListMeetings). The specific effort tools differ by domain.

## Per-Package Analysis

### Calendar

**File:** `packages/sage-benchmark/sage_benchmark/calendar_scheduling/evaluation/due_diligence/evaluate.py`
**File:** `packages/sage-benchmark/sage_benchmark/calendar_scheduling/evaluation/due_diligence/judge.py`

Calendar currently measures three sub-metrics, mixing deterministic tool-call counts with LLM-judged conversation analysis.

#### Current Implementation

`evaluate_due_diligence()` (line 60 of `evaluate.py`) computes:

1. **Message count** (line 85): `sum(1 for e in emails if e.from_ == task.assistant.email)` -- number of emails sent by the assistant. This is deterministic.

2. **Preference mention count** (lines 87-101): For each assistant email, an LLM judge (`DueDiligenceJudge.evaluate_preference_mention()`, line 82 of `judge.py`) determines whether the email references the assistant's time preferences. The prompt (lines 13-32 of `judge.py`) defines what constitutes a mention (explicit preferred time windows, scheduling constraints framed as preference) vs. not a mention (just proposing a time, discussing availability without preference framing). Returns `PreferenceMentionJudgement.mentions_preference: bool`.

3. **Proposal count** (lines 103-114): Sum of:
   - **Formal proposals** (`_count_formal_proposals()`, line 28): Counts `RequestMeeting` and `ReplyMeeting(COUNTER)` tool calls from the assistant context. These are the "unit of work" effort actions.
   - **Text proposals** (lines 107-113): For each `SendEmail` body, an LLM judge (`DueDiligenceJudge.evaluate_text_proposal()`, line 114 of `judge.py`) checks whether the email contains a specific time-based proposal (e.g., "How about 3pm?"). The prompt (lines 34-52 of `judge.py`) distinguishes proposals from availability statements, questions, or confirmations.

The result is stored as `DueDiligenceResult` with fields `message_count`, `preference_mention_count`, and `proposal_count` (lines 16-25).

#### Alignment with Decision

The decision specifies effort actions = `RequestMeeting` + `ReplyMeeting(COUNTER)`. The current `proposal_count` partially aligns (it counts these via `_count_formal_proposals()`), but also includes text-based proposals detected by LLM judge. The decision would simplify this to just the tool-call count.

The `message_count` and `preference_mention_count` metrics are supplementary and would not be the primary DD metric under the decision, though they could remain as secondary metrics.

### Form-Filling

**File:** `packages/sage-benchmark/sage_benchmark/form_filling/evaluation/due_diligence/evaluate.py`

Form-filling currently measures ask_user appropriateness (P/R/F1) and masked field accuracy. This will change significantly after AskUser removal (see 1.5).

#### Current Implementation

`evaluate_due_diligence()` (line 71) computes:

1. **Ask-user P/R/F1** (lines 201-214): Based on `user_qa_history`:
   - TP = asked unfindable field (correct to ask user since info is not in artifacts)
   - FP = asked findable field (wrong to ask user since info is available in artifacts)
   - FN = unfindable not asked (should have asked user but did not)
   - Precision = TP / (TP + FP), Recall = TP / (TP + FN), F1 = harmonic mean

2. **Masked field accuracy** (lines 197-199): `sum(correct) / len(field_evals)` -- fraction of masked fields answered correctly. This measures outcome, not effort.

3. **Question-field mapping** (lines 119-153): Each ask_user question is mapped to form fields using an LLM judge (`QuestionFieldMappingJudgment`). This determines which fields each question covers.

#### Alignment with Decision

The decision redefines form-filling DD as: count of `Search*` + `Read*` tool calls (SearchEmail, ReadEmail, SearchCalendar, ReadCalendar). This is a fundamentally different metric:
- **Current:** Measures whether the agent asked the right questions to the user.
- **Proposed:** Measures whether the agent performed sufficient file-system searches to gather information independently.

After AskUser removal (1.5), the ask_user P/R/F1 metrics become undefined. The new DD metric would count file-search tool calls from the agent's tool history. This requires:
1. Extracting tool calls from `assistant_context` (the message history).
2. Counting calls to SearchEmail, ReadEmail, SearchCalendar, ReadCalendar.
3. Optionally normalizing by the number of masked fields to get a "search effort per field" ratio.

### Marketplace

**File:** `packages/sage-benchmark/sage_benchmark/marketplace/evaluation.py` (lines 6-31)

Marketplace measures due diligence via `_compute_role_metrics()`, counting messages, offers, and price range explored.

#### Current Implementation

`_compute_role_metrics()` (line 6 of `evaluation.py`) iterates over `exec_result.action_trace` for a given role and computes:

1. **message_count** (line 15): Count of `SendMessage` actions by this role.
2. **offer_count** (line 17): Count of `MakeOffer` actions by this role.
3. **price_range_explored** (lines 23-25): `(max(prices) - min(prices)) / zopa` when there are 2+ offers and zopa > 0. This measures how much of the negotiation space the agent explored.

These are stored in `RoleMetrics` (lines 149-152 of `types.py`) and computed separately for buyer and seller (lines 71-72 of `evaluation.py`).

#### Alignment with Decision

The decision specifies effort action = `MakeOffer` only. The current `offer_count` already measures this exactly. The `message_count` and `price_range_explored` are supplementary metrics.

- `message_count` counts `SendMessage`, which is a communication action but not a substantive "unit of work" (it does not move the negotiation forward in the same way as an offer).
- `price_range_explored` is a derived metric from the offer prices, measuring exploration breadth. It could be retained as a secondary metric.
- Under the decision, the primary DD metric for marketplace would be `offer_count` (already computed), potentially with `price_range_explored` as a quality modifier.

## Cross-Benchmark Summary

| Benchmark | Decision: Effort Actions | Current Primary DD Metric | Gap |
|---|---|---|---|
| Calendar | `RequestMeeting` + `ReplyMeeting(COUNTER)` | proposal_count (formal + LLM-detected text proposals) | Drop text proposal detection; use only tool-call count |
| Form-Filling | `Search*` + `Read*` | ask_user P/R/F1 + masked field accuracy | Completely new metric after AskUser removal |
| Marketplace | `MakeOffer` | offer_count | Already aligned; just promote to primary metric |
