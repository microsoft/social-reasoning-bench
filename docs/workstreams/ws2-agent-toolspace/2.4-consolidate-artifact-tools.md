# 2.4 Consolidate Form-Filling Artifact Tools

## Definition & Architectural Decision

Replace the four form-filling artifact tools (`SearchEmail`, `ReadEmail`, `SearchCalendar`, `ReadCalendar`) with two generic tools: `SearchFiles(query)` and `ReadFile(id)`. Artifacts are returned as files with `.eml` (email) and `.ics` (calendar) extensions, making the file type visible to the agent without requiring separate tool names per artifact type.

**Rationale:**
- The email/calendar split is artificial — both are just documents in the persona's private artifact store
- The BM25 index already maintains separate indices internally, but the agent doesn't need to know this
- A unified `SearchFiles` can search across both email and calendar simultaneously, improving recall
- `ReadFile` returns the full content regardless of type — the extension tells the agent what it's looking at
- Reduces the tool count from 4 → 2, simplifying the agent's action space
- Aligns conceptually with calendar's artifact access (which doesn't split by artifact type — `ListMeetings` returns everything)

## Per-Package Analysis

### Calendar

**Not affected.** Calendar's artifact access tools (`ListMeetings`, `ListContacts`) already operate on a unified view — there's no email/calendar split. The agent calls `ListMeetings` and gets all meetings regardless of type.

### Form-Filling

**Current state:** Four tools defined in `form_filling/environment/actions.py`:

```python
class SearchEmail(Tool):
    """Search emails by keyword query. Returns top 3 results with snippets."""
    query: str

class ReadEmail(Tool):
    """Read the full content of an email by its ID."""
    id: str

class SearchCalendar(Tool):
    """Search calendar events by keyword query. Returns top 3 results with snippets."""
    query: str

class ReadCalendar(Tool):
    """Read the full content of a calendar event by its ID."""
    id: str

FILE_SYSTEM_TOOLS = [SearchEmail, ReadEmail, SearchCalendar, ReadCalendar]
```

The `BM25Index` in `form_filling/environment/bm25_index.py` maintains separate indices (`_email_bm25` and `_cal_bm25`) and has separate methods (`search_emails()`, `search_calendar()`, `get_email()`, `get_calendar_event()`). The `execute_tool()` dispatcher routes by tool name.

**Target state:** Two tools:

```python
class SearchFiles(Tool):
    """Search the user's files by keyword query. Returns top results with snippets."""
    query: str

class ReadFile(Tool):
    """Read the full content of a file by its ID."""
    id: str

FILE_SYSTEM_TOOLS = [SearchFiles, ReadFile]
```

**Changes required:**
1. `form_filling/environment/actions.py` — replace 4 tool classes with 2
2. `form_filling/environment/bm25_index.py` — add unified `search_all(query)` that searches both email and calendar indices, merges and ranks results. Add file extension to IDs (e.g., `email_001.eml`, `cal_003.ics`). Update `execute_tool()` dispatcher.
3. `form_filling/agents/assistant.py` — update tool list references
4. `form_filling/one_shot.py` — update tool registration
5. `form_filling/interactive.py` — update tool registration
6. `form_filling/evaluation/due_diligence/evaluate.py` — update tool name matching for effort counting (was `SearchEmail`/`ReadEmail`/etc., now `SearchFiles`/`ReadFile`)
7. `form_filling/prompts.py` — update any prompt text referencing the old tool names

**BM25 merge strategy:** When `SearchFiles` is called, search both `_email_bm25` and `_cal_bm25`, collect scored results from both, sort by combined score, return top-k. This is a simple merge — the BM25 scores are comparable since both indices use the same tokenizer and parameters.

**File extension mapping:**
- `artifact_type == "email"` → ID becomes `{id}.eml`
- `artifact_type == "calendar"` → ID becomes `{id}.ics`
- `ReadFile` strips the extension to look up the artifact by original ID

### Marketplace

**Not affected.** Marketplace has no artifact access tools — the agent's only private information is the reservation price in its instruction prompt.
