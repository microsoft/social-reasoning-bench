# 3.6 Run Mode Separation

## Definition & Architectural Decision

All benchmarks must support `--run-mode {all,exec,eval,reeval}` to allow independent control over which phases run:

- **`all`**: Execute tasks and then evaluate results (default).
- **`exec`**: Execute tasks only, skip evaluation.
- **`eval`**: Evaluate previously saved execution results, skip execution.
- **`reeval`**: Re-evaluate an existing complete output file (useful for testing new evaluation logic against old runs).

---

## Per-Package Analysis

### Calendar

**File:** `packages/sage-benchmark/sage_benchmark/calendar_scheduling/cli.py`

Calendar does not use a single `--run-mode` flag. Instead, it provides three separate flags that achieve the same effect through different mechanics:

- `--restart-exec` (line 366): `action="store_true"`. When used with `--resume`, re-runs execution from scratch while preserving the checkpoint directory.
- `--restart-eval` (line 360): `action="store_true"`. When used with `--resume`, re-runs evaluation from scratch using checkpointed execution results.
- `--reeval` (line 378): `type=Path`. Points to an existing output JSON file and runs evaluation only.

The default behavior (no flags) is equivalent to `--run-mode all`.

**Mapping to target:**
| Target mode | Calendar equivalent |
|---|---|
| `all` | Default (no flags) |
| `exec` | Not directly supported; closest is running without judge model |
| `eval` | `--resume <path> --restart-eval` |
| `reeval` | `--reeval <path>` |

**Gap:** Calendar lacks an explicit `exec`-only mode. The `--restart-exec` and `--restart-eval` flags work only with `--resume`. Consolidating these into a `--run-mode` flag would simplify the interface.

### Form-Filling

**File:** `packages/sage-benchmark/sage_benchmark/form_filling/cli.py` (lines 60-64)

Form-filling has the closest match to the target:

```python
parser.add_argument(
    "--run-mode",
    choices=["all", "tasks", "eval"],
    default="all",
)
```

- `all`: Runs tasks + evaluation.
- `tasks`: Runs tasks only, skips evaluation.
- `eval`: Loads prior results from `--task-results-path` and runs evaluation only.

There is no `reeval` mode. The `eval` mode requires `--task-results-path` (line 66) pointing to a `task_results.json` file.

**Gap:** Rename `tasks` to `exec` for consistency. Add `reeval` mode. The `--task-results-path` should be subsumed by the checkpoint system (see 3.3).

### Marketplace

**Files:**
- `packages/sage-benchmark/sage_benchmark/marketplace/cli.py`

Marketplace has no `--run-mode` support. The `main()` function (line 70) always runs execution followed by evaluation:

```python
results = asyncio.run(run_tasks(...))
evaluations = [evaluate_task(r) for r in results]
```

There is no way to:
- Run execution without evaluation
- Re-evaluate previous results
- Resume from saved execution results

**Gap:** Add `--run-mode {all,exec,eval,reeval}` flag to `marketplace/cli.py` with the same semantics as the target. This depends on checkpoint support (3.2) for the `eval` and `reeval` modes.
