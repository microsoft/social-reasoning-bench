# 3.2 Checkpoint/Resume for Marketplace

## Definition & Architectural Decision

Marketplace gets checkpoint/resume capability modeled on calendar's system. This allows long-running negotiation experiments to be interrupted and resumed without re-running completed tasks. The checkpoint file stores execution results, evaluation results, and the run configuration so that a resumed run can pick up exactly where it left off.

---

## Per-Package Analysis

### Calendar (Reference Implementation)

**Files:**
- `packages/sage-benchmark/sage_benchmark/calendar_scheduling/checkpoints/manager.py`
- `packages/sage-benchmark/sage_benchmark/calendar_scheduling/checkpoints/data.py`
- `packages/sage-benchmark/sage_benchmark/calendar_scheduling/checkpoints/run_config.py`

The calendar checkpoint system consists of three components:

1. **`CheckpointData`** (`data.py`, line 11): A Pydantic model storing:
   - `config: RunConfig | None` -- full CLI configuration for resume
   - `metadata: BenchmarkMetadata | None` -- run metadata
   - `source_file_hashes: dict[str, str]` -- SHA256 hashes of source YAML files for integrity checks
   - `execution_results: list[TaskExecutionResult]` -- completed executions
   - `evaluation_results: list[TaskEvaluationResult]` -- completed evaluations
   - `completed_task_keys: list[str]` and `completed_eval_keys: list[str]` -- quick lookup lists
   - `last_updated: str` -- ISO timestamp
   - `interrupted: bool` -- whether the run was interrupted

2. **`CheckpointManager`** (`manager.py`, line 16): Manages save/load operations:
   - `load()` (line 32): Reads and validates checkpoint file, populates internal key sets.
   - `save()` (line 56): Atomic write via temp-file-then-rename.
   - `initialize()` (line 68): Creates a new checkpoint with config, metadata, and source hashes.
   - `add_execution_result()` (line 90): Appends result and saves.
   - `add_evaluation_result()` (line 100): Appends result and saves.
   - `set_interrupted()` (line 111): Marks run as interrupted.
   - `get_completed_task_keys()` / `get_completed_eval_keys()` (lines 117-123): For filtering already-done work.
   - `cleanup()` (line 143): Removes checkpoint file after successful completion.

3. **`RunConfig`** (`run_config.py`, line 15): Stores all CLI options as a Pydantic model. Includes `from_args()` (line 109) to create from CLI namespace and `from_metadata()` (line 147) to reconstruct from saved metadata.

The CLI (`cli.py`) supports `--resume` (line 347), `--force-resume` (line 354), `--restart-eval` (line 360), and `--restart-exec` (line 366) flags.

### Marketplace (Current State)

**Files:**
- `packages/sage-benchmark/sage_benchmark/marketplace/cli.py`
- `packages/sage-benchmark/sage_benchmark/marketplace/runner.py`

Marketplace has no checkpointing. The CLI (`cli.py`) has no `--resume` flag. The `run_tasks()` function in `runner.py` (line 157) is a simple sequential loop that writes all results to a single `results.json` file at the end of the run (lines 122-138). If the run is interrupted, all progress is lost.

**Gaps to address:**

1. **Create `marketplace/checkpoints/` package** mirroring calendar's structure:
   - `data.py` with `CheckpointData` containing marketplace-specific result types (`TaskExecutionResult`, `TaskEvaluationResult` from `marketplace/types.py`).
   - `manager.py` with `CheckpointManager` providing save/load/resume.
   - `run_config.py` with `RunConfig` capturing marketplace CLI options.

2. **Add CLI flags** to `marketplace/cli.py`:
   - `--resume` with optional path argument
   - `--restart-eval` for re-evaluation
   - `--output-dir` already exists (line 49)

3. **Integrate into `runner.py`**: Pass `CheckpointManager` to `run_tasks()` and call `add_execution_result()` after each task.

4. **Task keying**: Marketplace tasks need content-based keys (SHA256 of task JSON) for checkpoint matching, similar to calendar's `compute_task_key()`.
