# 3.3 Checkpoint/Resume for Form-Filling

## Definition & Architectural Decision

Form-filling gets checkpoint/resume capability to replace its current ad-hoc append-to-JSON approach. The current system offers partial crash resilience (results are saved incrementally) but lacks the ability to skip already-completed tasks on resume, validate source file integrity, or separate execution from evaluation phases.

---

## Per-Package Analysis

### Calendar (Reference Implementation)

See **3.2** for the full calendar checkpoint reference. Key components:
- `CheckpointManager` with atomic save, task-key-based skip sets, and `interrupted` flag.
- `RunConfig` capturing all CLI options for exact resume.
- `CheckpointData` with separate lists for execution and evaluation results.
- CLI flags: `--resume`, `--force-resume`, `--restart-eval`, `--restart-exec`.

### Form-Filling (Current State)

**File:** `packages/sage-benchmark/sage_benchmark/form_filling/runner.py`

Form-filling uses `append_batch_to_json_list()` (line 50) for incremental saving:

```python
def append_batch_to_json_list(path: Path, items: list[dict]):
    if path.exists():
        with open(path) as f:
            data = json.load(f)
    else:
        data = []
    data.extend(items)
    with open(path, "w") as f:
        json.dump(data, f, indent=2)
```

This function is called after each batch completes, writing to separate files:
- `task_results.json` (line 189): Execution results
- `eval_results.json` (line 190): Evaluation results
- `summary.json` (line 191): Final summary

The CLI (`cli.py`) provides a basic `--run-mode` flag (line 60) with choices `["all", "tasks", "eval"]` and a `--task-results-path` flag (line 66) for loading prior results in eval-only mode. There is also a `_load_prior_interactive_results()` helper (line 75) for reconstructing execution results from a saved file.

**Issues with the current approach:**

1. **No skip logic**: If a run is interrupted mid-batch, resuming will re-run all tasks from the beginning. The `append_batch_to_json_list` function appends to the file but there is no mechanism to detect which tasks were already completed.

2. **Not atomic**: The read-modify-write cycle in `append_batch_to_json_list` is not crash-safe. If the process dies during the write, the file may be corrupted.

3. **No config persistence**: Unlike calendar's `RunConfig`, the form-filling run parameters are not saved. Resuming requires the user to re-specify all CLI arguments.

4. **No source integrity checks**: No file hashing to detect if the input data changed between runs.

**Gaps to address:**

1. **Create `form_filling/checkpoints/` package** with:
   - `CheckpointData` storing `TaskExecutionResult` / `InteractiveTaskExecutionResult` and evaluation results.
   - `CheckpointManager` with atomic save (temp-file-then-rename) and task-key-based skip sets.
   - `RunConfig` capturing form-filling CLI options.

2. **Add task keying**: Form tasks need content-based keys for checkpoint matching (e.g., SHA256 of `form_id`).

3. **Add CLI flags**: `--resume` with optional path, `--restart-eval`, `--restart-exec`.

4. **Replace `append_batch_to_json_list`**: Use `CheckpointManager.add_execution_result()` after each task (not after each batch) for finer-grained progress tracking.

5. **Consolidate `--run-mode`**: The existing `--run-mode` choices (`all`, `tasks`, `eval`) partially overlap with the checkpoint system's `--restart-eval` / `--restart-exec`. These should be unified to match the shared convention (see 3.6).
