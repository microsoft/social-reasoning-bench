# 3.4 Shared BenchmarkLogger with --log-style Flag

## Definition & Architectural Decision

All benchmarks must use the shared `BenchmarkLogger` ABC from `shared/logging.py` with a `--log-style {verbose,progress,quiet}` CLI flag (or `--logger` as currently named in calendar). This replaces ad-hoc `print()` and `tqdm` usage with a structured logging abstraction that can be swapped at runtime.

---

## Per-Package Analysis

### Shared (Reference Implementation)

**File:** `packages/sage-benchmark/sage_benchmark/shared/logging.py`

The logging module provides:

1. **`BenchmarkLogger` ABC** (line 11): Abstract base class with six hooks:
   - `on_phase_start(phase, total_items)` (line 20)
   - `on_task_start(task_id)` (line 30)
   - `on_task_round(task_id, round_idx, max_rounds)` (line 39)
   - `on_task_complete(task_id, success, error)` (line 50)
   - `on_phase_complete(phase, completed, failed)` (line 61)
   - `log_message(level, message, *args)` (line 72)
   - Context manager support (`__enter__`/`__exit__`) for progress bar lifecycle.

2. **`VerboseLogger`** (line 91): Emits `logger.info()` for every task/round. Preserves original CLI-style output.

3. **`ProgressLogger`** (line 122): Uses `tqdm` progress bar with dynamic postfix updates. Writes errors below the bar via `tqdm.write()`.

4. **`QuietLogger`** (line 192): Only logs errors and final phase summaries. Useful for CI.

5. **`create_benchmark_logger(style)`** (line 228): Factory function mapping `"verbose"`, `"progress"`, or `"quiet"` to the corresponding class.

### Calendar

**Files:**
- `calendar_scheduling/cli.py` (lines 219-223): `--logger` flag with choices `["verbose", "progress", "quiet"]`.
- `calendar_scheduling/runner.py` (line 11): Imports `BenchmarkLogger`, accepts it as parameter (line 222).
- `calendar_scheduling/evaluation/evaluator.py` (lines 18, 33, 121): Accepts `BenchmarkLogger` for evaluation phase tracking.
- `calendar_scheduling/experiments/runner.py` (lines 14, 397, 610): Passes `BenchmarkLogger` through experiment infrastructure.

Calendar uses `BenchmarkLogger` throughout its execution and evaluation pipelines. The logger is created in `cli.py` (line 644) via `create_benchmark_logger(args.logger)` and threaded into the `Experiment` runner.

**Status:** Fully compliant.

**Note:** The CLI flag is named `--logger` rather than `--log-style`. A minor rename may be desirable for consistency across benchmarks, but the functionality is complete.

### Form-Filling

**File:** `packages/sage-benchmark/sage_benchmark/form_filling/runner.py`

Form-filling uses bare `print()` statements extensively (over 80 occurrences in `runner.py`) and `tqdm.asyncio.tqdm` (imported at line 12) in some code paths. There is no `BenchmarkLogger` usage and no `--log-style` / `--logger` CLI flag.

Examples of ad-hoc logging:
- `print(f"Loaded {len(tasks)} form tasks from {data_path}")` (line 201)
- `print(f"Running {len(tasks)} tasks in one-shot mode (batches of {batch_size})")` (line 342)
- `print(f"Task {task_idx}: FAILED - {str(result)}")` (line 389)
- Summary output (lines 1086-1145): Long block of `print()` calls for metrics.

**Gap:** Replace all `print()` calls with `BenchmarkLogger` hooks. Add `--log-style` flag to `form_filling/cli.py`. The summary output (lines 1086-1145) should be factored into a `print_evaluation_summary()` function analogous to calendar's.

**Gap:** Adopt `BenchmarkLogger` in `runner.py` (replace `on_task_complete` callback with logger hooks) and `cli.py` (replace print statements). Add `--log-style` flag to `parse_args()`.
