# 4.2 Judge Votes

## Definition & Architectural Decision

Standardize judge vote count across all benchmarks. A single default is used, exposed as `--judge-votes` on the CLI. Multiple independent LLM judges evaluate each privacy leakage candidate, and a majority-vote rule determines the final verdict.

---

## Per-Package Analysis

### Calendar

**CLI flag:** `--judge-votes` in `calendar_scheduling/cli.py` (line 242-246), default **3**.

The value flows through `RunConfig.judge_votes` (`checkpoints/run_config.py` line 87-90, default 3) into the evaluator. In `evaluation/evaluator.py`, `evaluate_single_task()` (line 32) accepts `judge_votes: int = 3` and passes it as `n_judges` to the privacy evaluator (line 50).

The underlying leakage judge (`evaluation/privacy/leakage/judge.py` line 87) has `DEFAULT_N_JUDGES = 5` as its class-level default, but the evaluator overrides this with the CLI-provided value (default 3).

**Inconsistency:** The `evaluation/privacy/evaluate.py` function (line 21) defaults `n_judges=5`, but when called from the evaluator, the CLI's `judge_votes=3` takes precedence. If `evaluate_privacy` is called directly (e.g., from the privacy CLI at `evaluation/privacy/cli.py` line 374), it uses whatever is passed.

### Form-Filling

**No `--judge-votes` CLI flag.** The vote count is hardcoded at the function level:

- `FormFillingLeakageJudge.DEFAULT_N_JUDGES = 5` (`evaluation/privacy/leakage/judge.py` line 42)
- `evaluate_privacy()` defaults `n_judges=5` (`evaluation/privacy/evaluate.py` line 24)
- `evaluate_conversation_privacy()` defaults `n_judges=5` (`evaluation/privacy/evaluate.py` line 84)

The runner passes no override, so form-filling always uses 5 judges.

**Differences from Calendar:**
- No CLI exposure; the value is not configurable at runtime.
- Default is 5 (vs calendar's effective default of 3 via `--judge-votes`).

### Marketplace

**No judge-based privacy evaluation.** Marketplace measures privacy via post-hoc reservation-price inference probes (`runner.py` lines 31-38): after the negotiation ends, each agent is asked to estimate the opponent's reservation price. The `buyer_inference_error` and `seller_inference_error` in `TaskEvaluationResult` (`types.py` lines 159-160) are simple numerical deltas, not majority-voted LLM judgments. No `n_judges` or `judge_votes` parameter exists anywhere in the marketplace package.
