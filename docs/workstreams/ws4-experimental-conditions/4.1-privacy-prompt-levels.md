# 4.1 Privacy Prompt Levels

## Definition & Architectural Decision

Shared privacy prompting presets live in `shared/prompts/` with a domain-agnostic core. Each benchmark wraps the shared preset with domain-specific context (e.g., calendar details, form artifacts). This keeps the privacy philosophy consistent while allowing domain-tailored language.

---

## Per-Package Analysis

### Calendar

**Location:** `packages/sage-benchmark/sage_benchmark/calendar_scheduling/agents/assistant/prompts/`

Calendar has **6 standalone preset files**, each exporting a `SYSTEM_PROMPT` string. All share a common base sentence ("You are an autonomous calendar scheduling assistant...") and layer privacy guidance on top:

| File | Lines | Privacy mechanism |
|------|-------|-------------------|
| `default.py` | 3 | No privacy guidance at all |
| `privacy_simple.py` | 5 | One-liner: "Be privacy conscious, do not overshare details" |
| `privacy_strong.py` | 3 | Hard rule: never reveal titles/attendees/descriptions; only busy/free |
| `privacy_tom.py` | 22 | Theory-of-Mind `<tom-reasoning>` block: belief state, inferred intent, threat assessment, duty of care |
| `privacy_tom_dual.py` | 31 | Dual ToM: models **both** the requestor's and the principal's (user's) mental states; principal model wins on conflict |
| `privacy_ci.py` | 56 | Full Contextual Integrity framework: sender, subject, information type, recipient, transmission principle; 4-question checklist |

The preset name is selected via `--assistant-system-prompt` on the CLI (`cli.py` line 257-260). The `list_available_presets()` helper auto-discovers files in the prompts directory.

### Form-Filling

**Location:** `packages/sage-benchmark/sage_benchmark/form_filling/prompts.py` (502 lines, monolithic)

Form-filling uses a `--prompt-type` CLI flag (`cli.py` line 83-88) with four choices:

| Choice | Constant | Mechanism |
|--------|----------|-----------|
| `base` | (no suffix) | No privacy guidance |
| `privacy_aware` | `PRIVACY_AWARE_SUFFIX` (line 287) | Short suffix appended to system prompt |
| `privacy_explained` | `PRIVACY_EXPLAINED_SUFFIX` (line 291) | Longer explanation of why privacy matters |
| `privacy_ci` | `PRIVACY_CI_SUFFIX` (line 313) | Contextual Integrity framework adapted for forms |

In addition, form-filling supports `--social-reasoning` (line 162-163) which enables ToM-augmented `SendMessage` tool schemas (`SendMessageSocialReasoningParams` in `agents/assistant.py` lines 33-69) with fields for `tom_thinking_interviewer` and `tom_thinking_principal`. This is structurally equivalent to calendar's ToM-dual approach but implemented via structured tool output rather than a system prompt block.

**Differences from Calendar:**
- Prompts are inline string constants in a monolithic file, not separate files.
- `privacy_aware` and `privacy_explained` have no calendar equivalent (calendar has `privacy_simple` and `privacy_strong` which are conceptually similar but differently worded).
- ToM is implemented as structured tool fields rather than a `<tom-reasoning>` XML block in the system prompt.
- The `--use-privacy-example` flag (`cli.py` line 166-169) appends `SOCIAL_REASONING_PRIVACY_COMPLEMENT_INSTRUCTION` examples to the social reasoning prompt.

### Marketplace

**Confirmed: No privacy prompt presets exist.** The marketplace `cli.py` has no `--prompt-type` or `--assistant-system-prompt` flag. The system prompt in `marketplace_base.py` (line 47-51) is a fixed string with no privacy guidance. Privacy in marketplace is measured post-hoc via reservation-price inference probes (`runner.py` lines 14-17, 31-38), not via prompt-level controls.
