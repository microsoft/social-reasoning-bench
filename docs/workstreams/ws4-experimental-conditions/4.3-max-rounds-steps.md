# 4.3 Max Rounds & Steps

## Definition & Architectural Decision

All benchmarks expose `--max-rounds` and `--max-steps-per-turn` as CLI flags to control conversation length and per-turn tool-calling depth. This ensures experiments can be consistently parameterized across benchmarks.

---

## Per-Package Analysis

### Calendar

**Location:** `calendar_scheduling/cli.py` lines 149-159

Both flags are present:

| Flag | Default | Description |
|------|---------|-------------|
| `--max-rounds` | 20 | Maximum conversation rounds per task |
| `--max-steps-per-turn` | 20 | Maximum tool calls per agent turn |

Both values are stored in `RunConfig` (`checkpoints/run_config.py` lines 59-60) and flow through to `runner.py`'s `run_single_task()` (line 215-216). The main loop iterates up to `max_rounds` (line 315), and each `_run_agent_turn()` call (line 119) takes `max_steps` which caps tool calls per turn. If max steps is exceeded, the turn ends without error (line 163-164).

### Form-Filling

**Location:** `form_filling/cli.py` lines 92-97

| Flag | Default | Description |
|------|---------|-------------|
| `--max-rounds` | 50 | Maximum conversation rounds in interactive mode |
| `--max-gui-steps` | 30 | Maximum interaction steps per form in GUI mode (line 179-182) |

**Missing:** There is no `--max-steps-per-turn` flag. In interactive mode, each agent turn produces exactly one tool call (the `SendMessage`, `EndConversation`, or `AskUser` action), so steps-per-turn is effectively always 1. In GUI mode, `--max-gui-steps` serves a similar purpose but is a total step cap, not per-turn.

**Note:** `--max-rounds` only applies to interactive mode. One-shot mode has no rounds concept (it is a single LLM call). The `runner.py` function signature (line 122) accepts `max_rounds: int = 50`.

### Marketplace

**Location:** `marketplace/cli.py` lines 27-30

| Flag | Default | Description |
|------|---------|-------------|
| `--max-steps-per-turn` | 3 | Maximum tool calls per agent turn |

**Missing:** There is no `--max-rounds` CLI flag. Instead, `max_rounds` is defined **per task in the YAML data** as a field on `MarketplaceTask` (`types.py` line 59: `max_rounds: int = Field(default=12, ge=1)`). The runner reads `task.max_rounds` directly (line 107: `for round_num in range(1, task.max_rounds + 1)`).

**Difference from Calendar:** Calendar treats `max_rounds` as a global CLI parameter applying to all tasks; marketplace treats it as task-level configuration baked into the data. This means different marketplace tasks in the same run can have different round limits.
