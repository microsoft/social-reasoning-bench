# 5.9 Audit: GUI Mode

## Definition & Architectural Decision

Assess the maintenance cost of GUI mode in the form-filling package. GUI mode uses Playwright browser automation with a vision-based LLM to fill HTML forms, which is a fundamentally different execution path from one-shot and interactive modes.

---

## Per-Package Analysis

### Calendar

**Not applicable.** Calendar has no GUI mode. All interaction is tool-call-based via the `CalendarAgent` class. There is no browser automation, screenshot processing, or visual form rendering.

### Form-Filling

**GUI-specific files:**

| File | Lines | Purpose |
|------|-------|---------|
| `gui.py` | 562 | Core GUI runner: Playwright browser automation, screenshot capture, HTML form interaction |
| `gui_prompt.py` | ~410 | Builds GUI-specific system prompts using `qwen_helpers` for Fara-7B model |
| `qwen_helpers/schema.py` | vendored | Qwen message schema types |
| `qwen_helpers/fncall_prompt.py` | vendored | Function-call prompt templates |
| `qwen_helpers/base_tools.py` | vendored | Qwen tool definitions |
| `qwen_helpers/utils.py` | vendored | Qwen utilities |

**Estimated GUI-only code:** ~1,000+ lines across `gui.py`, `gui_prompt.py`, and the 4 `qwen_helpers` files.

**CLI footprint:** `form_filling/cli.py` has 4 GUI-specific arguments:

| Flag | Lines | Description |
|------|-------|-------------|
| `--execution-mode gui` | 17-21 | Mode selector (shared with one-shot, interactive) |
| `--http-port` | 172-176 | HTTP port for serving forms (default: 8080) |
| `--max-gui-steps` | 178-182 | Maximum interaction steps per form (default: 30) |
| `--restructure-model` | 184-189 | Model for restructuring HTML values to Pydantic (default: `trapi/msraif/shared/gpt-4.1`) |

**Architecture overview** (from `gui.py` header, lines 1-10):
- Uses a vision-based LLM agent (e.g., Fara-7B) to fill HTML forms
- Playwright headless browser renders the form
- Agent sees screenshots and issues click/type actions
- Extracted HTML values are restructured into the Pydantic form schema via a separate LLM call
- Prerequisites: `pip install playwright && playwright install`

**Dependencies unique to GUI mode:**
- `playwright` (browser automation)
- `qwen_helpers` vendored module
- HTTP server for serving form HTML
- Screenshot/base64 encoding for vision input

**Shared code used by GUI mode:**
- `prompts.py`: `format_artifacts_as_context`, `get_thinking_prompt`, `translate_persona_to_text`
- `schemas.py`: `FormTask`, `TaskExecutionResult`, `FormFillingAction`
- `agents/oracle_user.py`: `OracleUser`
- `environment/bm25_index.py`: `BM25Index`
- `runner.py`: Integrated via `run_single_fara_task` and `start_http_server` imports

**Runner integration:** `runner.py` imports GUI functions unconditionally (line 22), meaning `gui.py` is always imported even in one-shot/interactive mode. If `playwright` is not installed, this would cause import errors. The actual execution is gated by mode selection.

**Validation:** The CLI validates GUI mode separately (lines 208-210), auto-defaulting the assistant model to `microsoft/Fara-7B`.

**Maintenance cost assessment:**
- **High coupling:** GUI mode depends on Playwright browser internals, HTML form rendering, screenshot processing, and the vendored Qwen-Agent code -- all orthogonal to the core form-filling evaluation.
- **External dependency:** Playwright requires separate installation (`playwright install`) and browser binaries, adding CI/deployment complexity.
- **Vendored code:** `qwen_helpers` is copied from QwenLM/Qwen-Agent with no update mechanism.
- **Unique restructuring step:** GUI mode requires a second LLM call (`--restructure-model`) to convert raw HTML field values into the Pydantic schema, adding cost and potential error.
- **Import pollution:** Unconditional import of `gui.py` in `runner.py` means GUI dependencies affect all modes.

### Marketplace

**Not applicable.** Marketplace has no GUI mode. All interaction is tool-call-based via the `MarketplaceAgent` class. The negotiation is entirely text-based with no visual component.

**Recommendation paths:**
1. **Keep and isolate:** Move GUI mode into a separate optional subpackage with lazy imports, so `playwright` is only required when GUI mode is actually used.
2. **Deprecate:** If Fara-7B / vision-based form filling is no longer an active research direction, remove `gui.py`, `gui_prompt.py`, and `qwen_helpers/` entirely (~1,000+ lines).
3. **Hybrid:** Keep the restructuring step (useful for any vision-based agent) but replace `qwen_helpers` with direct prompt construction.
