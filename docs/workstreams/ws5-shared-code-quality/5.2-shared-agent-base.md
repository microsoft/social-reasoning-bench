# 5.2 Shared Agent Base Class

## Definition & Architectural Decision

A single LLM tool-calling agent base class in `shared/agent.py` provides the common pattern: maintain a message history, generate tool calls via LLM, parse and validate responses, handle retries on invalid output. Each benchmark subclasses this with domain-specific behavior.

---

## Per-Package Analysis

### Calendar

**Location:** `calendar_scheduling/agents/calendar_base.py` -- `CalendarAgent` (lines 33-301)

**Constructor** (lines 40-71):
- Accepts `model`, `model_client: ModelClient`, `allowed_contacts`, `tools`, `explicit_cot`
- Stores `_messages: list[ChatCompletionMessageParam]`
- Builds `_openai_tools` from tool classes
- Tracks `_previous_response_id` for thinking preservation across turns

**Message management:**
- `add_new_messages()` (line 78): Injects new emails by simulating a `GetEmails` tool call + tool response
- `add_tool_call_result()` (line 101): Appends tool result to last tool call
- `add_forced_action()` (line 119): Records a programmatic (non-LLM) action into history

**Tool call generation** -- `generate_tool_call()` (line 173):
- Retry loop with configurable `max_retries=3`
- Optional explicit CoT reasoning phase before tool call (lines 181-190)
- Validates exactly 1 tool call, correct type, known function name
- Validates `SendEmail` recipient against `allowed_contacts`
- On parse failure, appends error context and retries
- On success, appends assistant message with tool call to permanent history
- Raises `ExceptionGroup` after all retries exhausted

**Unique features:** Explicit CoT support, `SendEmail` recipient validation, `_previous_response_id` for thinking preservation.

### Form-Filling

**Location:** `form_filling/agents/assistant.py` -- `AssistantAgent` (lines 131-306)

**Constructor** (lines 134-195):
- Accepts `client`, `model`, `persona`, `artifacts`, `prompt_type`, `temperature`, `file_system`, `bm25_index`, `social_reasoning`, `use_privacy_example`
- Stores `messages: list[ChatCompletionMessageParam]` (public, not private)
- Builds tools via `_build_tools()` helper, optionally adding file-system tools
- Initializes system prompt + artifacts context + instruction as first 3 messages

**Tool call generation** -- `generate_action()` (line 197):
- Returns `(tool_name, arguments)` tuple (not a `Tool` instance)
- No retry loop -- generates once and returns
- Handles file-system tools (`SearchEmail`, `ReadEmail`, etc.) internally by executing them and recursing (lines 226-255)
- Only returns "terminal" actions (`SendMessage`, `EndConversation`, `AskUser`) to the caller
- Records `SendMessage` content + thinking fields into message history
- Tracks ToM intent history for cross-turn social reasoning (lines 272-284)

**Message management:**
- `add_tool_result()` (line 298): Appends as user message
- `add_new_message()` (line 301): Appends interviewer message as user message, optionally prepending intent history

**Differences from Calendar:**
- No `Tool` object returned; returns raw `(name, arguments)` tuple
- No retry logic; single LLM call per action
- No `_previous_response_id` tracking
- Recursive file-system tool handling (auto-executes search/read tools internally)
- Uses `tool_choice="required"` instead of `tool_choice="auto"`
- Temperature support via kwargs

### Marketplace

**Location:** `marketplace/agents/marketplace_base.py` -- `MarketplaceAgent` (lines 23-171)

**Constructor** (lines 26-55):
- Accepts `role`, `model`, `model_client`, `instruction_message`
- Stores `_messages: list[ChatCompletionMessageParam]`
- Initializes with system message + user instruction
- Builds `_openai_tools` from `MARKETPLACE_TOOLS`
- Tracks `_previous_response_id`

**Message management:**
- `add_turn_marker()` (line 61): Injects round/deadline awareness message
- `add_new_messages()` (line 74): Simulates `GetMessages` tool call (same pattern as calendar's `add_new_messages`)
- `add_tool_call_result()` (line 97): Appends tool result

**Tool call generation** -- `generate_tool_call()` (line 105):
- Retry loop with `max_retries=3`
- Validates exactly 1 tool call, correct type, known function name
- On parse failure, appends user message with error (not tool message)
- Returns parsed `Tool` instance
- Raises `RuntimeError` after all retries (vs calendar's `ExceptionGroup`)

**Extra:** `generate_text_response()` (line 163): Calls model without tools for post-negotiation probes.

### Comparison Summary

| Feature | Calendar | Form-Filling | Marketplace |
|---------|----------|--------------|-------------|
| Message storage | `_messages` (private) | `messages` (public) | `_messages` (private) |
| Return type | `Tool` instance | `(str, dict)` tuple | `Tool` instance |
| Retry logic | Yes (3 retries) | No | Yes (3 retries) |
| Error on exhaustion | `ExceptionGroup` | `ValueError` (single) | `RuntimeError` |
| `tool_choice` | `"auto"` | `"required"` | `"auto"` |
| CoT support | Yes (explicit) | No (structured thinking) | No |
| Response ID tracking | Yes | No | Yes |
| Email injection pattern | `GetEmails` simulation | Direct user message | `GetMessages` simulation |
| Recipient validation | Yes (`allowed_contacts`) | No | No |
| Temperature control | No | Yes | No |
| Internal tool execution | No | Yes (file-system tools) | No |

**Consolidation path:** Calendar and marketplace agents share the most structure (message management, retry loop, `_previous_response_id`). Form-filling's agent is more divergent due to its recursive file-system tool handling, lack of retries, and different return type. A shared base could extract: message history, `add_new_messages` simulation pattern, retry-aware `generate_tool_call`, and response ID tracking.
