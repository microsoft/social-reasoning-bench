# 5.3 Error Handling

## Definition & Architectural Decision

All benchmarks classify errors as fatal (unrecoverable, stop task) vs retriable (transient, continue) to maintain resilience during long multi-task runs. A fatal error aborts the current task but does not crash the entire benchmark.

---

## Per-Package Analysis

### Calendar

**Location:** `calendar_scheduling/runner.py` lines 35-112 -- `_is_fatal_error()`

Calendar has the most sophisticated error classification with a dedicated 77-line function:

**Fatal error types checked (in order):**
1. `litellm.ContextWindowExceededError` -- context overflow
2. `litellm.AuthenticationError` -- 401 invalid API key
3. `litellm.NotFoundError` -- 404 model not found
4. `litellm.BudgetExceededError` -- proxy budget constraint
5. `openai.AuthenticationError` -- 401 (unwrapped)
6. `openai.PermissionDeniedError` -- 403 no access
7. `openai.NotFoundError` -- 404 (unwrapped)
8. `RateLimitError` with `insufficient_quota` or `exceeded your current quota` substring -- fatal quota issue (vs recoverable throttling)
9. String pattern matching for context length (`context_length_exceeded`, `maximum context length`)
10. Regex patterns for auth failures (lines 94-106): `authenticat`, `unauthorized`, `invalid.*key`, `token.*expired`, `access.*denied`, `credential`, `login.*required`, `permission.*denied`, `forbidden`

**Recoverable errors:** Everything else (rate limits from throttling, timeouts, network issues, service unavailable).

**Usage in runner:** Both assistant and requestor turns (lines 340-347, 370-377) catch exceptions. Fatal errors set `fatal_error` string and break the loop. Recoverable errors log a warning and end the turn, advancing to the next round.

The `fatal_error` field is stored on `TaskExecutionResult` (types.py line 311-314) and the `is_valid` computed property (line 317-319) returns `False` if a fatal error occurred.

### Form-Filling

**Location:** `form_filling/runner.py` (no `_is_fatal_error` function)

Grepping for `_is_fatal_error`, `fatal_error`, `fatal`, and `recoverable` across the entire `form_filling/` package returns **zero matches**.

Error handling is simpler: the runner uses `asyncio` batch processing with `tqdm`. Individual task failures are caught by `try/except` in the batch processing functions and logged, but there is no fatal-vs-retriable classification. Failed tasks produce error entries in the results but do not propagate exceptions that would crash the run.

The `one_shot.py` module has a `MAX_RETRIES = 3` constant (line 35) for structured output parsing, but this is a parse-level retry, not an API error retry.

**Gap:** Form-filling has no mechanism to distinguish between a temporary rate limit (should retry) and an auth failure (should stop). All errors are treated uniformly.

### Marketplace

**Location:** `marketplace/runner.py` lines 41-78 -- `_run_agent_turn()`

Marketplace handles errors at the turn level:

```python
try:
    action = await agent.generate_tool_call()
except Exception:
    action_trace.append({
        "action_type": "GENERATION_ERROR",
        "result": traceback.format_exc(),
        "valid": False,
    })
    return 1, True  # ended=True
```

When `generate_tool_call()` raises any exception:
- The error is recorded in the action trace as `GENERATION_ERROR`
- The turn is treated as ended (`True`)
- The runner then checks if the environment outcome is already set; if not, it marks the agent as having ended due to error (lines 122-129)

**No fatal-vs-retriable distinction.** Any exception during tool call generation ends the agent's participation in the negotiation. The `MarketplaceAgent.generate_tool_call()` itself has a 3-retry loop for parse errors (`ValidationError`, `RetryException`) but API-level errors are not retried.

**Comparison Summary:**

| Feature | Calendar | Form-Filling | Marketplace |
|---------|----------|--------------|-------------|
| Fatal error classifier | Yes (77-line function) | No | No |
| Auth error detection | Yes (regex patterns) | No | No |
| Context overflow detection | Yes (exception + string) | No | No |
| Quota vs rate-limit split | Yes | No | No |
| Error stored on result | `fatal_error` field | Error in results dict | `GENERATION_ERROR` trace |
| Parse retry | 3 retries | 3 retries (one-shot) | 3 retries |
| API error retry | Implicit (recoverable) | None | None |

**Consolidation path:** Calendar's `_is_fatal_error()` is domain-agnostic -- it only inspects exception types and error strings, not calendar-specific concepts. It could be extracted to `shared/errors.py` and reused by all three benchmarks.
