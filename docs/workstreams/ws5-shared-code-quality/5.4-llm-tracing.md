# 5.4 LLM Tracing

## Definition & Architectural Decision

All benchmarks integrate the `sage_llm` tracer for debugging LLM calls. Tracing is opt-in via CLI flag, initialized once at startup, and traces are saved to the output directory on completion or interruption.

---

## Per-Package Analysis

### Calendar

**Location:** `calendar_scheduling/cli.py`

Calendar has full tracing integration:

- **Import:** `from sage_llm import get_tracer, save_traces` (line 10)
- **CLI flag:** `--llm-tracing` (lines 385-390), `action="store_true"`, default `False`
- **Initialization:** `get_tracer()` called in `run()` when `base_args.llm_tracing` is True (lines 552-553)
- **Save on completion:** `save_traces(traces_path)` called via `experiment.run_paths.get_traces_path()` (lines 687-689) when the run is cancelled via signal handler
- **Propagation:** The `llm_tracing` flag is passed through to `Experiment` (line 651) and `run_multiple` (line 507) for experiment mode

The tracer captures all LLM calls made through `sage_llm.ModelClient` across the entire process. Traces are saved alongside other outputs in the run directory.

### Form-Filling

**Confirmed: No tracing integration.** Grepping for `get_tracer` and `save_traces` across the entire `form_filling/` package returns zero matches.

Form-filling uses `sage_llm.ModelClient` for LLM calls (imported in `runner.py` line 10, `one_shot.py` line 8, `agents/assistant.py` line 7), so the underlying LLM client is compatible with tracing. However, there is no CLI flag to enable it, no `get_tracer()` call to initialize it, and no `save_traces()` call to persist results.

**Gap:** If a user needs to debug LLM calls in form-filling, they have no built-in mechanism. They would need to manually instrument the code or use external proxy logging.

### Marketplace

**Confirmed: No tracing integration.** Grepping for `get_tracer` and `save_traces` across the entire `marketplace/` package returns zero matches.

Like form-filling, marketplace uses `sage_llm.ModelClient` (imported in `cli.py` line 9, `runner.py` line 7, `agents/marketplace_base.py` line 13), so the client is tracer-compatible. But there is no initialization or persistence code.

**Comparison Summary:**

| Feature | Calendar | Form-Filling | Marketplace |
|---------|----------|--------------|-------------|
| CLI flag | `--llm-tracing` | None | None |
| `get_tracer()` init | Yes | No | No |
| `save_traces()` persist | Yes | No | No |
| Uses `sage_llm.ModelClient` | Yes | Yes | Yes |
| Trace output path | `run_paths.get_traces_path()` | N/A | N/A |

**Consolidation path:** Since all three packages already use `sage_llm.ModelClient`, adding tracing support only requires:
1. Adding `--llm-tracing` CLI flag
2. Calling `get_tracer()` at startup when enabled
3. Calling `save_traces(path)` on completion

This is a small, low-risk change for form-filling and marketplace.
