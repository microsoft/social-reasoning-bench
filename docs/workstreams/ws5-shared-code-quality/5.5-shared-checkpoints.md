# 5.5 Shared Checkpoint System

## Definition & Architectural Decision

A generic checkpoint system in `shared/checkpoints/` with type parameters allows any benchmark to save and resume long-running experiments. The system should handle execution results, evaluation results, and run configuration generically.

---

## Per-Package Analysis

### Calendar (Reference Implementation)

**Location:** `calendar_scheduling/checkpoints/` -- 4 files:

#### `run_config.py` (174 lines) -- `RunConfig(BaseModel)`
Stores all CLI parameters needed to reproduce a run:
- Task paths, limits
- Model names, base URLs, API versions (per-agent: assistant, requestor, judge)
- Reasoning effort (per-agent)
- `max_rounds`, `max_steps_per_turn`, `batch_size`
- System prompt, explicit CoT, artifacts, expose preferences
- `judge_votes`
- `output_dir`, `variant` (experiment-specific)
- `from_args()` classmethod constructs from `argparse.Namespace`
- `from_metadata()` classmethod reconstructs from saved `BenchmarkMetadata`
- Resolved model properties (`resolved_assistant_model`, etc.) handle fallback from per-agent to global

#### `data.py` (51 lines) -- `CheckpointData(BaseModel)`
The serialized checkpoint state:
- `config: RunConfig | None` -- the run configuration
- `metadata: BenchmarkMetadata | None` -- run metadata (models, timestamps)
- `source_file_hashes: dict[str, str]` -- SHA256 hashes of source YAML files for integrity checking
- `execution_results: list[TaskExecutionResult]` -- completed executions
- `evaluation_results: list[TaskEvaluationResult]` -- completed evaluations
- `completed_task_keys: list[str]` -- task keys that finished execution
- `completed_eval_keys: list[str]` -- task keys that finished evaluation
- `last_updated: str` -- ISO timestamp
- `interrupted: bool` -- whether run was cancelled

#### `manager.py` (148 lines) -- `CheckpointManager`
Manages checkpoint lifecycle:
- `__init__(checkpoint_path, enabled)` -- path-based, can be disabled
- `load()` -- reads and validates JSON checkpoint file
- `save()` -- atomic write via temp file + rename (line 64-66)
- `initialize(config, metadata, source_file_hashes)` -- create fresh checkpoint
- `add_execution_result(result)` -- append + save
- `add_evaluation_result(result)` -- append + save
- `set_interrupted(bool)` -- mark run as cancelled
- `get_completed_task_keys()` / `get_completed_eval_keys()` -- set lookups for skip logic
- `cleanup()` -- removes checkpoint file after successful completion

#### `hashing.py`
Provides content-based task keys (SHA256 of task JSON) for deterministic checkpoint matching.

### What is Calendar-Specific vs Generic

**Calendar-specific (would need type parameters):**
- `CheckpointData` references `TaskExecutionResult` and `TaskEvaluationResult` from `calendar_scheduling.types`
- `RunConfig` contains calendar-specific fields: `assistant_system_prompt`, `expose_preferences`, `explicit_cot`, `requestor_*` fields, `artifacts`
- `from_args()` and `from_metadata()` assume calendar's CLI structure

**Already generic:**
- `CheckpointManager` logic: load/save/atomic-write/add-result/interrupt handling
- The checkpoint pattern: keyed tasks, separate exec/eval tracking, interrupted flag
- `hashing.py`: content-based key generation

### Form-Filling

**No checkpoint system.** Form-filling's `runner.py` uses `append_batch_to_json_list()` (lines 50-66) for incremental result saving, which provides some crash resilience but not true resumability -- there is no skip logic, no config preservation, and no evaluation checkpoint. If a run crashes mid-batch, completed task results are preserved in the JSON file, but the runner has no mechanism to detect which tasks were already done and skip them on restart.

### Marketplace

**No checkpoint system.** Marketplace runs sequentially (`runner.py` line 168: `for task in tasks`) with no incremental saving at all. Results are written to disk only after all tasks complete (cli.py line 138). A crash loses all progress.

**Consolidation path:** Extract `CheckpointManager` with generic type parameters `[TExecResult, TEvalResult, TConfig]`. The atomic save, load, key tracking, and interruption logic are fully reusable. Each benchmark would provide its own `TConfig` (replacing `RunConfig`) and result types. The `hashing.py` module is already generic.
