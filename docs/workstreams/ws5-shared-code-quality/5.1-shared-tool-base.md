# 5.1 Shared Tool Base Class

## Definition & Architectural Decision

A single `Tool(BaseModel)` class in `shared/tool.py` replaces the three near-identical copies currently maintained in each benchmark package. All domain-specific tools subclass this shared base.

---

## Per-Package Analysis

### Calendar

**Location:** `calendar_scheduling/types.py` lines 16-53

```python
class Tool(BaseModel):
    model_config = {"extra": "forbid"}

    @classmethod
    def get_name(cls) -> str: ...
    @classmethod
    def get_description(cls) -> str: ...
    @classmethod
    def get_parameters_schema(cls) -> dict[str, Any]: ...
    @classmethod
    def get_openai_function_tool_param(cls): ...
```

- `ToolError` exception defined at line 10.
- `get_parameters_schema()` pops `$defs` and `title` from the JSON schema. Has a special case: if `properties == {}`, returns `{"type": "object", "properties": {}}` (lines 39-40) with a comment about Gemini and Anthropic API compatibility.
- `get_openai_function_tool_param()` returns a typed `ChatCompletionFunctionToolParam` using OpenAI SDK types (lines 45-53).

### Form-Filling

**Location:** `form_filling/environment/actions.py` lines 14-45

```python
class Tool(BaseModel):
    model_config = {"extra": "forbid"}
    # Same 4 classmethods
```

- `get_parameters_schema()` is **identical** logic: pops `$defs`, `title`, handles empty properties.
- `get_openai_function_tool_param()` returns a **plain dict** (lines 37-45) rather than the typed `ChatCompletionFunctionToolParam`. The structure is the same but uses `dict` instead of SDK types:
  ```python
  return {
      "type": "function",
      "function": { "name": ..., "description": ..., "parameters": ... },
  }
  ```
- No `ToolError` exception in this file (form-filling does not use tool execution errors the same way).

### Marketplace

**Location:** `marketplace/types.py` lines 13-44

```python
class Tool(BaseModel):
    model_config = {"extra": "forbid"}
    # Same 4 classmethods
```

- `get_parameters_schema()` is **identical** to calendar's implementation.
- `get_openai_function_tool_param()` returns a typed `ChatCompletionFunctionToolParam` (like calendar), with an explicit return type annotation `-> ChatCompletionFunctionToolParam` (line 36).
- `ToolError` exception defined at line 9 with a different docstring: "Raised when a tool action violates simulation rules" (vs calendar's "Raised when a tool execution fails due to invalid input or state").

### Comparison Summary

| Feature | Calendar | Form-Filling | Marketplace |
|---------|----------|--------------|-------------|
| `model_config` | `{"extra": "forbid"}` | `{"extra": "forbid"}` | `{"extra": "forbid"}` |
| `get_name()` | `cls.__name__` | `cls.__name__` | `cls.__name__` |
| `get_description()` | `cls.__doc__ or ""` | `cls.__doc__ or ""` | `cls.__doc__ or ""` |
| `get_parameters_schema()` | Pops `$defs`/`title`, empty-props guard | Identical | Identical |
| `get_openai_function_tool_param()` | `ChatCompletionFunctionToolParam` | Plain `dict` | `ChatCompletionFunctionToolParam` |
| `ToolError` | Yes (same file) | No (not in this file) | Yes (same file) |

**Consolidation path:** The three `Tool` classes are functionally identical. The only real difference is form-filling's `get_openai_function_tool_param()` returning a dict instead of the typed SDK object. A shared base in `shared/tool.py` could unify these, with the dict-vs-typed difference resolved by standardizing on the typed return (or adding a `to_dict()` convenience method).
