# 6.5 Validation

## Definition & Architectural Decision

All pipelines should validate generated data before outputting. Validation should catch structural errors, semantic invariants, and coverage gaps before the data is written to final output files. Pipelines should either halt on validation failure or exclude invalid items with a report.

---

## Per-Package Analysis

### Calendar

**Files:**
- `packages/sage-data-gen/sage_data_gen/calendar_scheduling/validate.py`
- `packages/sage-data-gen/sage_data_gen/calendar_scheduling/verify.py`

Calendar has two validation layers: invariant verification (step 6) and output validation (step 11).

#### verify.py -- Invariant Verification (Step 6)

`verify_tasks()` (line 39) checks four invariants per task and **filters out** failing tasks:

1. **Satisfiable tasks must have free slots and a conflict** (lines 57-62):
   - `_count_free_slots()` must return > 0
   - `_has_any_conflict()` must return True (unless fully open with 11 free slots)

2. **Unsatisfiable tasks must have no free slots and a conflict** (lines 63-67):
   - Free slots must be 0
   - Must have a conflict with the requested meeting

3. **Conflict event must be secret** (lines 70-73):
   - `_get_conflict_event()` finds the conflicting meeting
   - That meeting must have `is_secret=True`
   - Skipped for fully open calendars (no conflict by design)

4. **Request must be at suboptimal preference time** (lines 76-85):
   - The requested meeting's time slot must NOT be at peak preference score
   - Ensures the privacy/duty-of-care tradeoff is meaningful

**Behavior on failure:** Failing tasks are excluded from the output (line 97: `verified.append(task)` only for passing tasks). A detailed report is saved to `_pipeline_outputs/6_verification.yaml` listing all failures with their issues (line 102-109).

#### validate.py -- Output Validation (Step 11)

`validate_output()` (line 8) performs structural assertions on the final YAML files:

1. **Non-empty task list** (line 12)
2. **Per-task checks** (lines 14-39):
   - `task.type == "calendar"`
   - `task.requestor.is_malicious is False` (ensures benign pipeline output)
   - `len(task.assistant.calendar) > 0` (non-empty calendar)
   - `len(task.assistant.contacts) > 0` (non-empty contacts)
   - All events have valid 5-char time strings (HH:MM format)
   - All events have `is_movable=False`
   - Requested meeting must conflict with at least one calendar event (unless fully open)
3. **Artifact validation** (lines 41-45): If artifacts file exists, checks that every task ID has corresponding artifacts.

**Behavior on failure:** Uses Python `assert` statements -- raises `AssertionError` and halts.

**Status:** Thorough two-layer validation. Structural and semantic checks. Failing tasks are excluded, not repaired.

### Marketplace

**File:** `packages/sage-data-gen/sage_data_gen/marketplace/validate.py`

`validate_tasks()` (line 6) performs three checks:

1. **Unique task IDs** (lines 8-10):
   - Checks that `len(ids) == len(set(ids))`
   - Error message: "Duplicate task IDs detected"

2. **Non-negative reservation prices** (lines 12-14):
   - Both `seller.reservation_price` and `buyer.reservation_price` must be >= 0

3. **Positive ZOPA (Zone of Possible Agreement)** (lines 15-20):
   - `buyer.reservation_price > seller.reservation_price`
   - This ensures a deal is theoretically possible -- the buyer is willing to pay more than the seller's minimum

**Return format:** A dict with `{"validation": {"total_tasks", "passed", "errors", "checks"}}`. The `checks` sub-dict has booleans for each category (`unique_ids`, `reservation_nonnegative`, `positive_zopa`).

**Behavior on failure:** In `cli.py` line 87, the pipeline raises `ValueError` if `validation["validation"]["passed"]` is False. This is a hard gate -- no tasks are written.

**Status:** Aligned but minimal. Only three checks. No semantic validation of instruction content, no check for story-price consistency, no validation that reservation stories actually contain the numeric values they claim to.

### Form-Filling

**Files:**
- `packages/sage-data-gen/sage_data_gen/form_filling/stages/validate_artifacts.py`
- `packages/sage-data-gen/sage_data_gen/form_filling/stages/validate_filesystem_artifacts.py`

Form-filling has the most sophisticated validation, using LLM-based checks and iterative repair.

#### validate_artifacts.py -- LLM-Based Artifact Validation (Stage 6)

`validate_artifacts_with_llm()` (line 40) uses an LLM judge to check three coverage dimensions:

1. **Field coverage** (close-ended fields):
   - Each ground truth field is checked for presence in artifacts
   - Fields are classified as open-ended or close-ended via `field_analysis`
   - Coverage metric: `covered_close_ended / total_close_ended`
   - The persona is included as a virtual "persona artifact" (index 0) for coverage checking

2. **Secret coverage**:
   - Each individual secret is checked for embedding in at least one artifact
   - Secrets are flattened to a list with unique IDs (`Q{i}_S{j}`)
   - Coverage metric: `embedded_secrets / total_secrets`
   - The `contains_secrets` tags on artifacts are rebuilt from validation results (lines 154-168)

3. **Negative info coverage**:
   - Each negative info item is checked for embedding
   - Coverage metric: `neg_embedded / neg_total`
   - Discards hallucinated coverage results when no negative info was provided (lines 171-172)

**Repair mechanisms:**

- `ensure_full_secret_coverage()` (line 385): Iterative loop (up to `max_secret_retries`) that generates new artifacts for uncovered secrets, then re-validates. Stops if no progress is made.
- `fix_missing_fields()` (line 285): Creates a single "Notes to self" artifact containing all missing close-ended field values.
- `fix_missing_negative_info_in_artifacts()` (line 542): Weaves missing negative info points into existing artifacts by scoring each artifact for compatibility and regenerating with the point included.

After repair, uncovered secrets are dropped entirely (generate_form_task.py, lines 219-244).

#### validate_filesystem_artifacts.py -- BM25 Retrievability Validation

`validate_bm25_retrievability()` (line 131) validates that findable fields' answer artifacts are retrievable via BM25 search:

1. **Corpus construction**: Each artifact's metadata + content is tokenized (custom tokenizer with stopword removal, line 86)
2. **BM25 indexing**: Uses `BM25Okapi` from the `rank_bm25` library
3. **Per-field validation**: For each findable field, tests all suggested search terms and checks if the answer artifact appears in top-k results (configurable via `config.bm25_top_k`, default 3)
4. **Output**: `BM25ValidationResult` with per-field pass/fail and overall pass rate

**Behavior on failure:** BM25 validation is informational -- results are saved to `bm25_validation.json` but do not block output generation.

**Status:** The most thorough validation of the three pipelines. Uses LLM-based semantic validation, iterative repair, and information retrieval validation. However, it is also the most expensive (multiple LLM calls per validation pass).

**Cross-pipeline comparison:**

| Aspect | Calendar | Marketplace | Form-Filling |
|--------|----------|-------------|--------------|
| Structural checks | Yes (assert) | Yes (dict report) | No (relies on Pydantic) |
| Semantic invariants | Yes (4 invariants) | Yes (3 checks) | Yes (LLM-based) |
| Coverage validation | No | No | Yes (field, secret, negative info) |
| Repair on failure | No (exclude) | No (halt) | Yes (iterative) |
| Retrieval validation | No | No | Yes (BM25) |
| Hard gate | verify: exclude; validate: assert | raise ValueError | No hard gate |
