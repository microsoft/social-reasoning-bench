# 6.6 Batch Generation

## Definition & Architectural Decision

All pipelines should support generating a full dataset in one run with concurrency. The pipeline should accept configuration for the total number of items to generate and use async concurrency to parallelize LLM calls where possible.

---

## Per-Package Analysis

### Calendar

**File:** `packages/sage-data-gen/sage_data_gen/calendar_scheduling/cli.py`

Calendar generates the entire dataset in a single `run_pipeline()` invocation with heavy use of `asyncio.gather()` for parallelism:

**Concurrent steps:**

1. **Step 2 -- Employee generation** (line 81):
   ```python
   employee_results = await asyncio.gather(
       *[generate_employees(client, company, config) for company in companies]
   )
   ```
   Parallelizes across companies (default 4 companies).

2. **Step 3 -- Calendar generation** (lines 100-105):
   ```python
   base_calendar_results = await asyncio.gather(*base_calendar_coros)
   ```
   Parallelizes across all employees (default 4 companies x 5 employees = 20 concurrent LLM calls).

3. **Step 4 -- Task generation** (line 161):
   ```python
   task_results = await asyncio.gather(*task_coros)
   ```
   Parallelizes across all (employee x archetype) pairs (20 employees x 7 archetypes = 140 concurrent coroutines). Each coroutine itself makes 3+ LLM calls (requestor generation, privacy labeling with 3 models, instruction generation).

**Concurrency control:** No explicit concurrency limit (no `asyncio.Semaphore`). All coroutines are launched simultaneously via `asyncio.gather()`. This relies on the `ModelClient` to handle rate limiting.

**Scale:** Default configuration generates 20 employees x 7 archetypes = 140 tasks, pruned by verification. A single run produces large/medium/small YAML files.

**Status:** Fully aligned. Full dataset generated in one run with aggressive parallelism.

### Marketplace

**File:** `packages/sage-data-gen/sage_data_gen/marketplace/generate_reservation_contexts.py`

Marketplace uses bounded concurrency for parallel context generation:

**Concurrent steps:**

1. **Step 2 -- Reservation context generation** (line 92):
   ```python
   sem = asyncio.Semaphore(max_concurrency)

   async def _bounded(*, i: int, cat: CatalogEntry) -> ReservationContext:
       async with sem:
           return await _generate_one(...)

   coros = []
   for i in range(total_tasks):
       cat = catalog[i % len(catalog)]
       coros.append(_bounded(i=i, cat=cat))

   return await asyncio.gather(*coros)
   ```
   - Default: `max_concurrency=12` (from `config.py` line 14)
   - Default: `total_tasks=280` (from `config.py` line 7)
   - All 280 coroutines are created, but only 12 run concurrently via the semaphore.

**Other steps:** Step 1 (catalog generation) is handled by `generate_catalog()` but its concurrency model was not examined in this analysis. Steps 3-7 are synchronous (assembly, validation, stats, writing).

**Status:** Aligned. Uses semaphore-bounded concurrency for the most expensive step. The `max_concurrency` parameter is user-configurable.

### Form-Filling

**Files:**
- `packages/sage-data-gen/sage_data_gen/form_filling/cli.py`
- `packages/sage-data-gen/sage_data_gen/form_filling/generate_form_task.py`
- `packages/sage-data-gen/sage_data_gen/form_filling/common_form_batch_creation.py`

Form-filling has significant gaps in batch generation.

#### CLI (`cli.py`)

The CLI processes exactly one form per invocation:
```python
task_dir = generate_form_task(
    image_path=args.image,
    output_dir=args.output_dir,
    config=config,
    form_id=args.form_id,
)
```
- `--image` accepts a single file path
- No `--batch` or `--input-dir` argument
- No concurrency

#### Per-form pipeline (`generate_form_task.py`)

`generate_form_task()` is entirely synchronous. LLM calls within it use the synchronous `client.chat.completions.parse()` (not `aparse()`). The stages run sequentially:
1. Parse form image (1 LLM call)
2. Generate ground truth (1+ LLM calls)
3. Mask fields (1+ LLM calls for standalone analysis)
4. Expand persona (1 LLM call)
5. Generate secrets (1 LLM call per open-ended field)
6. Generate negative info (1 LLM call per suitable field)
7. Create artifacts (1 LLM call)
8. Validate artifacts (1+ LLM calls, with retry loops)
9. Generate HTML (optional, 1 LLM call)

Total: ~10-20 synchronous LLM calls per form.

#### Batch runner (`common_form_batch_creation.py`)

`run_batch()` (line 29) processes forms serially from a JSONL file:

```python
for i, entry in enumerate(tqdm(entries, desc="Generating form tasks")):
    ...
    task_dir = generate_form_task(
        image_path=tmp_path,
        output_dir=output_dir,
        config=config,
        form_id=str(form_id),
    )
```

- Serial loop with `tqdm` progress bar
- No `asyncio`, no threading, no multiprocessing
- Each form is processed completely before starting the next
- `start` and `limit` parameters for partial runs (lines 33-34)
- Error handling: continues on failure, collects success/fail counts
- Returns a summary dict with lists of succeeded/failed forms

**Gaps vs. decision:**
1. **No concurrency at the form level**: Each form is processed sequentially. For 60 forms with ~10-20 LLM calls each, this means 600-1200 serial LLM calls.
2. **No concurrency within a form**: All LLM calls within `generate_form_task()` are synchronous. Some stages (e.g., per-field secret generation) could be parallelized.
3. **No `async` entry point**: `generate_form_task()` does not use `async/await`, so even if a batch runner wanted to parallelize, it would need significant refactoring.
4. **No configuration for concurrency**: No `max_concurrency` or similar parameter.

**Recommendation:**
1. Convert `generate_form_task()` to async, using `aparse()` instead of `parse()` for LLM calls.
2. Parallelize within-form stages where possible (e.g., per-field secret generation, per-field negative info generation).
3. Add a concurrent batch runner using `asyncio.Semaphore` (similar to marketplace) to process multiple forms in parallel.
4. Add `--max-concurrency` CLI argument to control parallelism.
