# 6.1 Pipeline Structure

## Definition & Architectural Decision

All data-gen pipelines should follow a multi-step structure with debug artifacts saved per step. Each step in the pipeline should call a `_save_step()` helper that writes a numbered YAML file to a `_pipeline_outputs/` subdirectory, enabling offline inspection and debugging of intermediate results. The calendar and marketplace pipelines already follow this pattern; form-filling does not.

---

## Per-Package Analysis

### Calendar

**File:** `packages/sage-data-gen/sage_data_gen/calendar_scheduling/cli.py`

The `run_pipeline()` function (line 63) implements a 12-step pipeline with explicit numbering, print statements, and `_save_step()` calls at key checkpoints:

| Step | Name | `_save_step()` call |
|------|------|---------------------|
| 1 | Generate companies | Line 76 -- saves company models |
| 2 | Generate employees | Line 86 -- saves per-company employee lists |
| 3 | Generate base full calendars | Line 111 -- saves 11-slot calendars per employee |
| 3b | Generate scheduling preferences | No `_save_step()` (lightweight deterministic mapping) |
| 4 | Generate + label tasks per archetype | Line 193 -- saves pre-assembly task counts and failure info |
| 5 | Assemble tasks (fullness + placement + trimming) | No `_save_step()` |
| 6 | Verify invariants | Line 226 -- saves verification report (passed/failed/issues) |
| 7 | Write `large.yaml` | Output file, not debug artifact |
| 8 | Write `medium.yaml` | Output file |
| 9 | Write `small.yaml` | Output file |
| 10 | Generate artifacts (optional) | External file, not debug step |
| 11 | Validate output | Uses `validate_output()`, no debug artifact saved |
| 12 | Summary stats | Line 269 -- saves stats dict |

The `_save_step()` helper (line 33) writes to `{output_dir}/_pipeline_outputs/{step}_{name}.yaml`. It uses `yaml.dump()` with `default_flow_style=False`.

**Concurrency:** Steps 2, 3, and 4 use `asyncio.gather()` for parallel LLM calls. Step 5 (assembly) and step 6 (verification) are deterministic and synchronous.

**Status:** Fully aligned with the decision.

### Marketplace

**File:** `packages/sage-data-gen/sage_data_gen/marketplace/cli.py`

The `run_pipeline()` function (line 47) implements a 7-step pipeline:

| Step | Name | `_save_step()` call |
|------|------|---------------------|
| 1 | Generate catalog | Line 59 -- saves catalog entries |
| 2 | Generate reservation contexts | Line 70 -- saves buyer/seller contexts |
| 3 | Assemble tasks | Line 76 -- saves first 10 assembled tasks as preview |
| 4 | Validate tasks | Line 85 -- saves validation report; raises `ValueError` on failure |
| 5 | Compute stats | Line 91 -- saves stats dict |
| 6 | Write `large.yaml` | Output file |
| 7 | Write `small.yaml` | Output file |

The `_save_step()` helper (line 22) is identical in pattern to calendar -- writes to `{output_dir}/_pipeline_outputs/{step}_{name}.yaml`. Uses `yaml.safe_dump()` (slightly different from calendar's `yaml.dump()`).

Note: Step 4 is a hard gate -- the pipeline halts if validation fails (line 87: `raise ValueError`). Calendar's validation (step 11) uses assertions but does not halt the pipeline.

**Status:** Fully aligned with the decision.

### Form-Filling

**Files:**
- `packages/sage-data-gen/sage_data_gen/form_filling/cli.py`
- `packages/sage-data-gen/sage_data_gen/form_filling/generate_form_task.py`
- `packages/sage-data-gen/sage_data_gen/form_filling/stages/` (8 stage modules)

The form-filling pipeline is fundamentally different. It processes **one form at a time**, not a batch:

1. **CLI entry point** (`cli.py`, line 14): `main()` parses args and calls `generate_form_task()` for a single `--image` argument. There is no batch loop, no `_pipeline_outputs/` directory, and no `_save_step()` calls.

2. **Per-form pipeline** (`generate_form_task.py`, line 50): `generate_form_task()` runs 7 stages sequentially inside a single function:

| Stage | Name | Saves |
|-------|------|-------|
| 1 | Parse form image | `form_model.py` (line 107) |
| 2 | Generate ground truth | `ground_truth.json` (line 116) |
| 2b | Mask close-ended fields | `masked_ground_truth.json`, `masked_fields.json` (lines 131-138) |
| 3 | Expand persona + secrets + negative info | Intermediate data only saved as part of `task.json` |
| 4-5 | Create + validate artifacts | `artifacts.json` (line 298) |
| FS | Generate filesystem artifacts (optional) | `filesystem_artifacts.json`, `findability.json`, `bm25_validation.json`, `distractor_scenarios.json` |
| 6 | Build and save `task.json` | `task.json` (line 394) |
| 7 | Generate GUI HTML (optional) | `form_{id}.html` (line 403) |

3. **Stage modules** (`stages/` directory): Each stage is a separate Python module (`parse_form.py`, `fill_groundtruth.py`, `mask_groundtruth.py`, `generate_scenario.py`, `generate_artifacts.py`, `validate_artifacts.py`, `generate_filesystem_artifacts.py`, `validate_filesystem_artifacts.py`, `generate_gui_html.py`). This is a cleaner separation of concerns than calendar/marketplace, which inline everything in `cli.py`.

4. **Batch support** (`common_form_batch_creation.py`): The `run_batch()` function (line 29) iterates over a JSONL file and calls `generate_form_task()` per entry in a serial loop (lines 76-115). No parallelism. No `_pipeline_outputs/` debug directory.

**Gaps vs. decision:**
- No `_save_step()` calls -- intermediate artifacts (persona, secrets, field analysis) are not saved to disk until final `task.json` assembly. If the pipeline fails at stage 5, stage 3 outputs are lost.
- No numbered step tracking to a debug directory.
- The batch runner is serial with no concurrency.
- Each form produces its own directory (`form_{id}/`) which is a different paradigm from the numbered-step debug output.

**Recommendation:** Add a `_save_step()` helper that writes intermediate results to a `_debug/` subdirectory within each `form_{id}/` directory. This preserves the per-form directory structure while adding step-level debuggability. Consider adding `asyncio` support to the batch runner.
