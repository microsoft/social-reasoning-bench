# 6.8 Config

## Definition & Architectural Decision

All pipelines should use a Pydantic config model with consistent patterns. Common fields that every pipeline should support: model selection, output directory, random seed, and concurrency control. This enables a uniform CLI interface and makes configuration serializable and validatable.

---

## Per-Package Analysis

### Calendar

**File:** `packages/sage-data-gen/sage_data_gen/calendar_scheduling/config.py`

```python
class PipelineConfig(BaseModel):
    num_companies: int = 4
    employees_per_company: int = 5
    calendar_date: str = "2026-02-20"
    fullness_levels: list[int] = Field(default_factory=lambda: [0, 1, 3, 5, 7, 9, 11])
    medium_size: int = 10
    small_size: int = 3
    task_retry_limit: int = Field(default=3, description="...")
    model: str = "phyagi/gpt-5.2"
    labeling_models: list[str] = Field(default_factory=lambda: ["phyagi/gpt-5.2", "phyagi/gpt-5.1", "phyagi/gpt-4.1"], description="...")
    output_dir: str = "data/calendar-scheduling/final"
    generate_preferences: bool = Field(default=True, description="...")
    random_seed: int = Field(default=42, description="...")
    generate_artifacts: bool = Field(default=False, description="...")
```

**Fields:** 13 fields total.

| Category | Fields |
|----------|--------|
| Scale | `num_companies`, `employees_per_company`, `fullness_levels` |
| Model | `model`, `labeling_models` (multi-model for majority vote) |
| Output | `output_dir`, `medium_size`, `small_size` |
| Behavior | `calendar_date`, `task_retry_limit`, `generate_preferences`, `generate_artifacts` |
| Reproducibility | `random_seed` |
| Concurrency | None |

**CLI mapping** (`cli.py`, `parse_args()`, line 278): All fields are exposed as CLI arguments. `labeling_models` is parsed from a comma-separated string. The config is printed as JSON at startup (line 361).

**Notable patterns:**
- Uses `Field(description=...)` for documenting non-obvious fields
- Uses `Field(default_factory=lambda: ...)` for mutable defaults
- No concurrency control (relies on `asyncio.gather()` without limits)
- Multi-model support via `labeling_models` list

**CLI mapping** (`cli.py`, `main()`, line 14): Model fields are exposed as `--parsing-model`, `--generation-model`, `--validation-model`, `--vision-model`. Not all config fields have CLI arguments -- filesystem-related fields use defaults unless `--filesystem` is enabled.

**Notable patterns:**
- Most model fields per pipeline (4 vs. 2 for marketplace and 2 for calendar)
- `output_dir` is NOT in the config -- it is passed as a CLI argument directly to `generate_form_task()`. This is inconsistent with calendar and marketplace.
- No `max_concurrency` (pipeline is synchronous)
- Different model endpoint format: `trapi/msraif/shared/gpt-5.1` vs. calendar's `phyagi/gpt-5.2` and marketplace's `trapi/gpt-4.1`

## Cross-Pipeline Comparison

| Field | Calendar | Marketplace | Form-Filling |
|-------|----------|-------------|--------------|
| Config class | `PipelineConfig` | `MarketplacePipelineConfig` | `FormFillingConfig` |
| `output_dir` | In config | In config | CLI arg only |
| `random_seed` | In config (42) | In config (42) | In config (42) |
| Model fields | `model`, `labeling_models` | `catalog_model`, `context_model` | `parsing_model`, `vision_model`, `generation_model`, `validation_model` |
| `max_concurrency` | Not present | 12 | Not present |
| Split sizes | `medium_size`, `small_size` | `small_size` | Not present |
| Retry config | `task_retry_limit` | `max_retries_per_item` | `max_mask_retries`, `max_secret_retries` |
| `Field(description)` | Yes (some) | No | No |

**Inconsistencies to resolve:**
1. **`output_dir`:** Should be in every config model for serialization completeness.
2. **`max_concurrency`:** Only marketplace has it. Calendar should add it (currently unbounded). Form-filling needs it when async support is added.
3. **Model field naming:** No consistent pattern. Calendar uses `model` (singular). Marketplace uses `{step}_model`. Form-filling uses `{purpose}_model`. Consider a convention like `{stage}_model` or a nested `ModelConfig`.
4. **Model endpoint prefixes:** Different API routing prefixes (`phyagi/`, `trapi/`, `trapi/msraif/shared/`). These are deployment-specific and likely intentional, but worth documenting.
5. **Split sizes:** Form-filling has no split size fields because it has no split mechanism (see 6.7).
6. **CLI construction:** Calendar builds the config from parsed args with a dict. Marketplace constructs directly with keyword args. Form-filling conditionally sets fields. Consider a unified pattern.
