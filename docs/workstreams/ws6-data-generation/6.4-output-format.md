# 6.4 Output Format

## Definition & Architectural Decision

Standardize output format across pipelines. Calendar and marketplace output YAML task files (`large.yaml`, `small.yaml`, etc.) containing a top-level `tasks` key with a list of task objects. Form-filling outputs per-task directories with multiple JSON files. Benchmark loaders must be able to consume the format each pipeline produces.

---

## Per-Package Analysis

### Calendar

**Data-gen output** (`packages/sage-data-gen/sage_data_gen/calendar_scheduling/cli.py`):

The `write_tasks_yaml()` function (line 40) serializes tasks to YAML:
```python
data = {"tasks": [task.model_dump(mode="json") for task in tasks]}
yaml.dump(data, f, default_flow_style=False, sort_keys=False, allow_unicode=True)
```

Output files:
- `large.yaml` -- all tasks (line 232), sorted by `free_slots_count` then `assistant.email`
- `medium.yaml` -- stratified subset via `stratified_subset()`, `config.medium_size` tasks per fullness level (line 235)
- `small.yaml` -- stratified subset, `config.small_size` tasks per fullness level (line 243)
- `large-artifacts.json` -- optional, generated in step 10 (line 252)
- `_pipeline_outputs/` -- debug artifacts per step

All YAML files share the same schema: `{"tasks": [CalendarTask, ...]}`.

**Benchmark loader** (`packages/sage-benchmark/sage_benchmark/calendar_scheduling/loader.py`):

`load_tasks()` (line 43) accepts a list of paths (files or directories). For each YAML file:
1. `yaml.safe_load()` expects `{"tasks": [...]}`
2. Each task dict is validated as `CalendarTask(**task_data)`
3. A content-based `task_key` hash is computed for checkpoint stability
4. Deduplication by task ID (within file and across files)
5. Optional `limit` parameter for subsetting

Artifacts are loaded separately via `load_artifacts()` (line 93) from a JSON file with `{"task_artifacts": [{"task_id": int, "artifacts": [...]}]}`.

**Status:** Clean, standardized format.

### Marketplace

**Data-gen output** (`packages/sage-data-gen/sage_data_gen/marketplace/cli.py`):

The `_write_tasks_yaml()` function (line 29) serializes tasks to YAML:
```python
payload = {"tasks": [task.model_dump(mode="json") for task in tasks]}
yaml.safe_dump(payload, f, sort_keys=False, allow_unicode=True)
```

Output files:
- `large.yaml` -- all tasks (line 94)
- `small.yaml` -- `_subset()` of first `config.small_size` tasks sorted by ID (line 97)
- No `medium.yaml` -- explicitly deleted if it exists (line 101: `_remove_if_exists()`)
- `_pipeline_outputs/` -- debug artifacts per step

Schema is identical to calendar: `{"tasks": [MarketplaceTask, ...]}`.

**Benchmark loader** (`packages/sage-benchmark/sage_benchmark/marketplace/loader.py`):

`load_tasks()` (line 38) follows the same pattern as calendar:
1. Accepts list of paths (files or directories)
2. `yaml.safe_load()` expects `{"tasks": [...]}`
3. Validates as `MarketplaceTask(**raw_task)`
4. Content-based `task_key` hash for checkpointing
5. Deduplication and optional `limit`

**Status:** Fully aligned with calendar's format.

### Form-Filling

**Data-gen output** (`packages/sage-data-gen/sage_data_gen/form_filling/generate_form_task.py`):

Each form produces a **directory** (`form_{id}/`) containing multiple files:

| File | Format | Contents |
|------|--------|----------|
| `form_model.py` | Python | Pydantic BaseModel class defining the form schema |
| `ground_truth.json` | JSON | `{field_id: {answer, is_open_ended}}` flat dict |
| `masked_ground_truth.json` | JSON | Ground truth with N fields blanked |
| `masked_fields.json` | JSON | List of `{field_id, original_value}` dicts |
| `artifacts.json` | JSON | `{"artifacts": [ArtifactDetail, ...]}` |
| `task.json` | JSON | Composite: persona, secrets, negative_info, validation, ground_truth answers, artifacts, coverage, due_diligence_fields |
| `form_{id}.html` | HTML | Interactive GUI form (optional, skipped with `--no-html`) |
| `image_{id}.{ext}` | Image | Copy of the source form image |
| `filesystem_artifacts.json` | JSON | File system artifacts for search evaluation (optional) |
| `findability.json` | JSON | Field findability classification (optional) |
| `bm25_validation.json` | JSON | BM25 retrievability validation (optional) |
| `distractor_scenarios.json` | JSON | Distractor scenarios per masked field (optional) |

**Key differences from YAML-based pipelines:**
1. **Per-task directories** instead of a single multi-task YAML file
2. **JSON format** instead of YAML
3. **Multiple files per task** instead of one serialized object
4. **Executable code** (`form_model.py`) as part of the task data
5. **No dataset-level aggregation** -- each form stands alone
6. **Redundancy:** `task.json` duplicates data from `artifacts.json` and `ground_truth.json`. For example, `task.json` contains `"artifacts": {"artifacts": [...]}` which is the same data as `artifacts.json`.

**Benchmark loader** (`packages/sage-benchmark/sage_benchmark/form_filling/loader.py`):

`load_single_form_task()` (line 21) reads from a task directory:
1. Loads `task.json`, `artifacts.json`, `ground_truth.json`, and `form_model.py`
2. Dynamically imports the Pydantic form class from `form_model.py`
3. Flattens `question_secrets` into a list of `SecretInfo` objects
4. Optionally loads `filesystem_artifacts.json`, `findability.json`, `distractor_scenarios.json`
5. Returns a `FormTask` object

`load_all_form_tasks()` (line 188) iterates over all `form_*/` subdirectories and loads each, sorted by form ID.

**Status:** Structurally different from calendar/marketplace. The directory-per-task approach has advantages (self-contained, includes executable code) but makes batch operations and dataset splitting harder.

**Recommendation:**
- Consider adding a `manifest.json` at the dataset level that lists all form task directories with metadata (form_id, split assignment, etc.). This would enable dataset-level operations (splits, subsetting) without loading every task.
- Evaluate whether the data in `task.json` can serve as the single source of truth, eliminating the redundancy with `artifacts.json` and `ground_truth.json`.
