# 6.7 Dataset Splits

## Definition & Architectural Decision

All benchmarks should output standardized dataset splits (large/small at minimum). Splits enable quick iteration during development (small) while providing statistical power for final evaluation (large). The split mechanism should be deterministic and preserve important dataset properties (e.g., stratification by difficulty level).

---

## Per-Package Analysis

### Calendar

**File:** `packages/sage-data-gen/sage_data_gen/calendar_scheduling/cli.py`

Calendar produces three splits: large, medium, and small.

#### Split mechanism

`stratified_subset()` (line 47) implements stratified sampling by fullness level:

```python
def stratified_subset(
    tasks: list[CalendarTask], n_per_level: int, fullness_levels: list[int]
) -> list[CalendarTask]:
    by_level: dict[int, list[CalendarTask]] = {lvl: [] for lvl in fullness_levels}
    for task in tasks:
        lvl = task.free_slots_count
        if lvl is not None and lvl in by_level:
            by_level[lvl].append(task)
    result = []
    for lvl in fullness_levels:
        result.extend(by_level[lvl][:n_per_level])
    return result
```

- Groups tasks by `free_slots_count` (0, 1, 3, 5, 7, 9, 11 by default)
- Takes the first `n_per_level` tasks from each group
- Deterministic: order depends on the sorted task list from step 6

#### Splits produced

| Split | File | Size formula | Default |
|-------|------|-------------|---------|
| Large | `large.yaml` | All verified tasks | ~140 (20 employees x 7 archetypes, minus verification failures) |
| Medium | `medium.yaml` | `medium_size` per fullness level x 7 levels | 10 x 7 = 70 |
| Small | `small.yaml` | `small_size` per fullness level x 7 levels | 3 x 7 = 21 |

- `medium_size` default: 10 (config.py line 9)
- `small_size` default: 3 (config.py line 10)
- `fullness_levels` default: `[0, 1, 3, 5, 7, 9, 11]` (config.py line 8)

After subsetting, tasks are re-sorted by task ID (lines 236, 244).

**Status:** Fully aligned. Three stratified splits with configurable sizes.

### Marketplace

**File:** `packages/sage-data-gen/sage_data_gen/marketplace/cli.py`

Marketplace produces two splits: large and small.

#### Split mechanism

`_subset()` (line 41) takes the first N tasks sorted by ID:

```python
def _subset(tasks: list[MarketplaceTask], n: int) -> list[MarketplaceTask]:
    if n >= len(tasks):
        return list(tasks)
    return list(sorted(tasks, key=lambda t: t.id)[:n])
```

- Simple truncation, no stratification
- Deterministic: sorted by task ID

#### Splits produced

| Split | File | Size | Default |
|-------|------|------|---------|
| Large | `large.yaml` | All tasks | 280 |
| Small | `small.yaml` | `small_size` | 21 |

- `small_size` default: 21 (config.py line 8)
- `total_tasks` default: 280 (config.py line 7)
- No `medium.yaml` -- explicitly deleted if it exists from a previous run (line 101: `_remove_if_exists(output_dir / "medium.yaml")`)

**No stratification:** Unlike calendar, marketplace does not stratify by any task property. The small split is simply the first 21 tasks by ID. This means the small split may not be representative of the full dataset's diversity (e.g., different product categories, price ranges, ZOPA sizes).

**Status:** Partially aligned. Two splits exist, but no stratification mechanism. A `medium.yaml` is intentionally excluded.

### Form-Filling

**Files:**
- `packages/sage-data-gen/sage_data_gen/form_filling/cli.py`
- `packages/sage-data-gen/sage_data_gen/form_filling/generate_form_task.py`
- `packages/sage-data-gen/sage_data_gen/form_filling/common_form_batch_creation.py`

Form-filling has no split mechanism.

#### Current state

- **CLI** (`cli.py`): Processes one form at a time. No concept of dataset-level splits.
- **Batch runner** (`common_form_batch_creation.py`): Processes a list of forms from JSONL and writes each to its own directory. No post-processing to create split assignments.
- **Benchmark loader** (`loader.py`): `load_all_form_tasks()` loads all `form_*/` directories in a given path. There is no `--split` parameter or split-aware loading.

Each form is independently generated and stored in `form_{id}/`. There is no mechanism to:
1. Assign forms to large/small splits
2. Stratify by form properties (number of fields, number of secrets, open-ended field count)
3. Create a subset manifest

#### How users currently select subsets

Users manually specify which form directories to include by pointing the benchmark at a directory containing only the desired forms, or by using the loader's optional `limit` parameter (though the form-filling loader does not expose this -- it loads all directories).

**Gaps:**
1. No dataset-level metadata file (e.g., `manifest.json`) that assigns forms to splits
2. No stratification by form properties
3. No CLI support for generating split assignments after batch creation
4. The benchmark loader has no split-aware loading

**Recommendation:**
1. After batch generation, add a post-processing step that creates a `splits.json` file mapping form IDs to split assignments (e.g., `{"large": ["form_350", "form_626", ...], "small": ["form_350", ...]}`).
2. Implement stratification by relevant form properties: number of fields, number of secrets, number of open-ended fields, form complexity.
3. Update `load_all_form_tasks()` to accept an optional `split` parameter that filters by the manifest.
4. Consider whether form-filling should also produce a `medium` split.
